# SQLite 知识体系(骨架)


## 一、基础认知
- SQLite 是什么:嵌入式、单文件、无服务端架构
- 核心特性:零配置、跨平台、单文件即数据库、动态弱类型
- 适用 vs 不适用场景
- 与 MySQL / PostgreSQL 的本质区别

### 一句话定义
**SQLite 是一个嵌入式、单文件、无服务端的关系型数据库。**

三个关键词拆开:

| 关键词 | 含义 |
|--------|------|
| **嵌入式(embedded)** | 数据库引擎不是一个独立服务,而是作为**库**直接嵌入到你的应用程序进程里,和应用跑在一起 |
| **单文件** | 整个数据库就是磁盘上的**一个文件**(通常 `.db` / `.sqlite`),没有一堆数据目录 |
| **无服务端(serverless)** | 不需要像 MySQL 那样先启动一个 `mysqld` 后台服务,程序直接读写那个文件 |

#### 架构对比(最核心的认知)

```
MySQL / PostgreSQL(C/S 架构):
  应用程序  ──网络/TCP──▶  数据库服务进程(mysqld)  ──▶  数据文件
                         (独立常驻,要启动、要配置)

SQLite(嵌入式架构):
  应用程序  ──直接调用 SQLite 库──▶  数据文件(.db)
           (没有中间服务,进程内直接读写文件)
```

> 这张图是理解 SQLite 一切特性的根源:**因为没有服务进程,所以零配置、单文件、不能跨机器共享、并发受限于文件锁**。后面所有特性都能从这张图推导出来。

#### 其他身份标签
- **自包含**:核心代码就一个 C 文件,无外部依赖
- **公共领域(public domain)**:比 MIT/BSD 还自由,可任意商用
- **部署最广的数据库**:每台手机(Android/iOS 默认)、每个浏览器、每架波音飞机的娱乐系统里都有它,全球部署量远超 MySQL


### 核心特性

#### 1. 零配置(zero-configuration)
- 不用装服务、不用配端口、不用建用户、不用改配置文件
- 装好 SQLite 库(或 Python 自带 `sqlite3` 模块),一行代码就能建库用
- 对比 MySQL:要 `mysqld` 启动、配 root 密码、开 3306 端口……

#### 2. 跨平台
- 一个 `.db` 文件可在 Windows / Mac / Linux / 手机之间**直接复制使用**,字节序统一
- 32 位和 64 位系统通用

#### 3. 单文件即数据库
- 整个库 = 一个文件,**复制文件 = 完整备份**
- 不像 MySQL 一个库对应一堆目录和 `.ibd` 文件,备份要专门工具
- 文件大小理论上支持 TB 级,但实际推荐控制在 **GB 级以内**性能最好

#### 4. 动态弱类型(SQLite 最独特的点)
- 列声明了类型,但你**可以存入任意类型的值**(声明 INTEGER 的列也能塞字符串)
- 靠"类型亲和(type affinity)"软约束,不是严格类型
- 这是 SQLite 和其他数据库最大的不同,后面"数据类型"那块会展开,**先记住"它不强制类型"**

#### 5. 无用户/权限系统
- 没有账号、没有角色、没有 `GRANT`/`REVOKE`
- 安全靠**操作系统文件权限**控制(谁能读这个 .db 文件,谁就能用)

#### 6. 标准事务支持
- 完整支持 ACID(原子性/一致性/隔离性/持久性)
- `BEGIN` / `COMMIT` / `ROLLBACK` 都支持,事务是 SQLite 的常规能力不是奢侈品

---

### 适用 vs 不适用场景

#### ✅ 适用场景

| 场景 | 例子 |
|------|------|
| **移动端/嵌入式本地存储** | 手机 App 的本地数据(Android/iOS 默认用 SQLite) |
| **桌面应用数据** | 浏览器书签/历史、各种软件的配置和缓存 |
| **单机工具/脚本** | CLI 工具、爬虫、数据分析脚本的本地数据存储(你那个 CLI TODO 就典型适用) |
| **原型/测试开发** | 早期不想搭 MySQL,用 SQLite 先跑起来,快速验证 |
| **中低并发网站** | 日访问量不大的站点、只读为主的站点 |
| **本地数据分析缓存** | 精算/数据分析师把数据落本地库做查询统计 |
| **配置/元数据存储** | 应用配置、版本元信息 |

#### ❌ 不适用场景

| 场景 | 原因 |
|------|------|
| **高并发写入** | SQLite 是"单写者"模型,大量并发写会因文件锁排队成瓶颈 |
| **多用户/跨机器共享的中央库** | 它是嵌入式,不能像 MySQL 那样多个客户端连同一个服务 |
| **海量数据 + 复杂查询** | 虽能 TB 级,但查询优化器不如 PostgreSQL 强,复杂 join 大数据下吃力 |
| **需要细粒度权限控制** | 没有用户/角色系统,做不了"某用户只能查某表" |
| **需要存储过程** | SQLite 没有存储过程 |

> 简单判断法:**数据要在"一台机器、一个程序"里用 → SQLite;要"多台机器、多个程序"共享 → MySQL/PostgreSQL**。

### 与 MySQL / PostgreSQL 的本质区别

| 维度 | SQLite | MySQL / PostgreSQL |
|------|--------|--------------------|
| **架构** | 嵌入式(无服务) | 客户端-服务端(C/S) |
| **部署** | 一个 .db 文件 | 服务进程 + 多个数据文件 |
| **配置** | 零配置 | 需装服务、配端口/用户/密码 |
| **访问方式** | 本机直接读写文件 | 支持网络远程连接 |
| **并发** | 单写多读(WAL 下读不阻塞写) | 多写多读,行级锁,高并发强 |
| **数据类型** | 动态弱类型(类型亲和) | 严格类型 |
| **用户权限** | 无 | 完整的用户/角色/权限体系 |
| **规模定位** | 单机、GB 级 | 多用户、TB+ 生产中央库 |
| **存储过程** | 无 | 有 |
| **备份** | 复制文件即可 | 需专用工具(mysqldump / pg_dump) |

#### 本质一句话
> **SQLite 是"库即文件、应用直连";MySQL/PostgreSQL 是"独立服务、网络访问"。**

理解了这句话,上面的对比表就都能推导出来:
- 因为"库即文件" → 单文件、零配置、复制即备份、靠文件权限
- 因为"应用直连" → 嵌入式、无网络、并发受限于文件锁
- 因为"独立服务" → 多客户端、高并发、有权限系统、要配置

## 二、数据类型与存储
- 5 种存储类:NULL / INTEGER / REAL / TEXT / BLOB
- 类型亲和(type affinity)机制
- 动态类型:列类型与存入值类型可不一致
  
SQLite 的类型系统是它和所有其他数据库最大的不同。普通数据库是"列声明什么类型就只能存什么",SQLite 则是**"声明归声明,存储归存储"**--列类型只是建议,实际存进去的值有自己的类型。理解这套机制要抓住三个概念:**存储类、类型亲和、动态类型**,它们是层层递进的关系。

### 5 种存储类(Storage Class):值本身的类型

存储类是**值在磁盘上实际存储时的类型**,SQLite 只有 5 种,这是最底层的真实分类:

- **NULL** -- 空值
- **INTEGER** -- 整数,1/2/3/4/6/8 字节,按值大小自动选
- **REAL** -- 浮点数,8 字节 IEEE 浮点
- **TEXT** -- 文本字符串,按数据库编码(UTF-8/UTF-16)存储
- **BLOB** -- 二进制大对象,原样存入,不解释

注意:存储类只有 5 种,但 SQLite 文档里常说的"数据类型"和"存储类"基本是一回事。其他数据库里熟悉的 `VARCHAR`、`DATE`、`BOOLEAN`、`DECIMAL` 这些类型 SQLite **并不真正区分**--比如 `DATE` 在 SQLite 里实际存为 TEXT 或 INTEGER,`BOOLEAN` 实际存为 INTEGER(0/1)。这点和 MySQL/PG 截然不同,后者每种类型有独立的存储格式。

### 类型亲和(Type Affinity):列的"倾向"

类型亲和是 SQLite 给"动态类型"加的一层软约束,用来在"列声明了类型"和"值可以任意类型"之间做平衡。它的意思是:**列并不强制某种类型,但有一个"倾向",存入的值会尽量往这个倾向转**。

SQLite 把列的亲和归为 5 大类,但实际只有 4 种有意义(NUMERIC 包含了 INTEGER/REAL 的混合行为):

| 亲和类型 | 含义 | 列声明示例 |
|----------|------|-----------|
| **TEXT** | 倾向存文本,存入的数值会被转成文本 | `CHAR`, `VARCHAR`, `TEXT`, `CLOB` |
| **NUMERIC** | 倾向存数值,文本里的数字会被转成 INTEGER/REAL | `NUMERIC`, `DECIMAL`, `BOOLEAN`, `DATE` |
| **INTEGER** | 等同 NUMERIC,但特例:整型主键会真正用 rowid | `INT`, `INTEGER`, `TINYINT`, `BIGINT` |
| **REAL** | 倾向存浮点,整数会被转成 REAL | `REAL`, `FLOAT`, `DOUBLE` |
| **BLOB**(无亲和) | 不做任何转换,存什么是什么 | 声明为 `BLOB` 或不声明类型 |

亲和的**判定规则**(SQLite 根据你建表时声明的类型字符串来推断亲和,而不是精确匹配):
- 声明里含 `INT` -> INTEGER 亲和
- 含 `CHAR`/`CLOB`/`TEXT` -> TEXT 亲和
- 含 `BLOB` 或**没声明类型** -> BLOB 亲和(无亲和)
- 含 `REAL`/`FLOA`/`DOUB` -> REAL 亲和
- 其他(含 `NUMERIC`/`DEC`/`DATE`/`BOOL` 等)-> NUMERIC 亲和

关键点:**亲和只是"尽量转",不是"必须转"**。比如一列声明 `INTEGER`,你存入字符串 `'abc'`(无法转成数字),SQLite 不会报错,而是**原样存为 TEXT**。只有能无损转换时才会转。这是亲和和严格类型的核心区别。

### 动态类型:列类型与存入值类型可不一致

这是前两个概念合起来的最终效果。普通数据库里,`CREATE TABLE t(x INTEGER)` 后插入 `INSERT INTO t VALUES('hello')` 会直接报错;SQLite 里却能成功,而且 `'hello'` 就以 TEXT 存储类躺在那个"INTEGER 列"里。

```sql
CREATE TABLE t(x INTEGER);
INSERT INTO t VALUES(1), (2.5), ('hello'), (NULL), (x'ABCD');
-- 全部成功,x 这一列里同时躺着 INTEGER、REAL、TEXT、NULL、BLOB 五种值
```

这就是"动态类型":**列声明的类型(经亲和判定)只是倾向,每个值实际按自己的存储类存放**。读出来时,SQLite 按值本身的类型处理,而不是按列类型。

这套机制的好处是**灵活**:同一列能存混合类型,适合存半结构化数据、配置、JSON 之类。坏处是**容易踩坑**:你以为某列都是数字,结果混进来一个字符串,排序、聚合、比较时行为会出乎意料(比如 `'10' < '9'` 因为按文本比,而 `10 < 9` 是 False)。所以工程实践中,虽然 SQLite 允许动态类型,但**自己应该保持列内类型一致**,把亲和当兜底而非主动利用。

### 三者的关系(一句话串起来)

存储类是**值的真实类型**(5 种,固定);类型亲和是**列的倾向**(4 种,由声明推断);动态类型是**最终效果**(值可以不服从列的倾向)。SQLite 没有"严格类型"这一层,亲和是唯一的软约束,能转就转、不能转就原样存--这就是它类型系统的全部。

## 三、SQL 基础语法
- **DDL**:create / alter / drop(库、表、索引、视图)
- **DML**:insert / update / delete
- **DQL**:select / where / order by / limit / group by / having
- **约束**:primary key / not null / unique / check / default / foreign key


### SQLite SQL 基础语法

SQL 按功能分四类,记住这个分类就能把所有语句归位:**DDL 定义结构、DML 改数据、DQL 查数据、约束管规则**。下面按这个顺序串起来,每个给最常用的语法和要点。

#### DDL:定义结构(create / alter / drop)

DDL 操作的是数据库的"骨架"--库、表、索引、视图的创建和销毁。

**库**:SQLite 里"建库"就是创建一个 `.db` 文件,没有 `CREATE DATABASE` 语句。打开一个不存在的库文件即自动创建,这是它"零配置"的体现。

**表**最常用。建表基本形态:

```sql
CREATE TABLE 表名 (
    列名 类型 约束,
    ...
    表级约束
);
```

`CREATE TABLE IF NOT EXISTS 表名 (...)` 加 `IF NOT EXISTS` 可避免重复创建报错。删表用 `DROP TABLE 表名`,改表用 `ALTER TABLE`,但 SQLite 的 `ALTER TABLE` 能力有限:**只支持改表名、加列(`ADD COLUMN`)**,不能直接删列、改列类型(老版本完全不行,新版 3.35+ 才支持 `DROP COLUMN`,但复杂改还是要重建表)。这是 SQLite 比 MySQL/PG 弱的地方,迁移表结构时会感到不便。

**索引**:为加速查询而建,`CREATE INDEX 索引名 ON 表名(列)`。索引会占空间、拖慢写入,不是越多越好,详细留到"索引"那块。

**视图**:把一条查询保存成"虚拟表",`CREATE VIEW 视图名 AS SELECT ...`。视图不存数据,每次查时动态执行底层 SQL,常用于简化复杂查询或做权限隔离。

#### DML:改数据(insert / update / delete)

DML 操作表里的"行",改的是数据不是结构。

**插入**:`INSERT INTO 表名(列1,列2) VALUES(值1,值2)`。省略列名时要按建表顺序给全部列值。批量插可用 `VALUES(...),(...),(...)`。SQLite 特有的 `INSERT OR REPLACE` 和 `INSERT OR IGNORE` 很实用:前者主键冲突时替换旧行,后者冲突时跳过--这是 SQLite 把"冲突处理"直接做进 INSERT 语法的小巧设计(等价于 UPSERT 的一种)。

**更新**:`UPDATE 表名 SET 列=值 WHERE 条件`。⚠️ **漏写 WHERE 会更新全表**,这是经典灾难,生产环境务必带条件。SQLite 的 UPDATE 不支持 `FROM`/`JOIN` 直接引用其他表(老版本),要基于别的表更新得用子查询。

**删除**:`DELETE FROM 表名 WHERE 条件`。同样**漏 WHERE 删全表**。想清空全表又想保留结构,`DELETE FROM 表名` 能用但慢(逐行删、记日志),更快的是 `DROP TABLE` 再重建,不过 SQLite 没有 `TRUNCATE` 语句(MySQL 有,SQLite 没有,这是个小差异)。

一个 SQLite 特色:`UPDATE`/`DELETE` 后可以用 `RETURNING` 子句(新版)直接返回被改的行,省去再查一次。

#### DQL:查数据(select)

DQL 是 SQL 的重头戏,`SELECT` 语句的执行顺序和书写顺序不同,理解这点是写好查询的关键。

书写顺序:`SELECT 列 FROM 表 WHERE 条件 GROUP BY 列 HAVING 条件 ORDER BY 列 LIMIT n`。

**实际执行顺序**:`FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT`。所以 WHERE 在分组前过滤行,HAVING 在分组后过滤组;SELECT 里起的别名 ORDER BY 能用,但 WHERE 用不了(因为还没执行到 SELECT)。这个顺序解释了 90% 的"为什么这条 SQL 报错"。

- **WHERE**:行级过滤,分组前执行,不能用聚合函数(如 `WHERE COUNT(*)>1` 会报错,该用 HAVING)。
- **GROUP BY**:按列分组,常配合聚合函数 `COUNT/SUM/AVG/MIN/MAX`。SQLite 比较宽松,允许 SELECT 出现非分组列(别的数据库会报错),但值取的是组内任意一行,不要依赖这个行为。
- **HAVING**:组级过滤,分组后执行,能用聚合函数。`HAVING COUNT(*)>1` 找重复组是经典用法。
- **ORDER BY**:排序,`ASC` 升序(默认)/`DESC` 降序。多列排序按声明顺序优先。
- **LIMIT**:限制返回行数,`LIMIT n` 或 `LIMIT n OFFSET m`(跳过 m 行)。SQLite 用 `OFFSET`,MySQL 也是,PostgreSQL 也兼容,语法基本通用。

DQL 的进阶(join、子查询、集合运算)留到"查询进阶"那块,这里先把单表查询的骨架打牢。

#### 约束:管规则的五件套

约束是建表时给列或表加的规则,用来保证数据质量。SQLite 支持五种核心约束,外加外键。

**PRIMARY KEY(主键)**:唯一标识一行,值不能重复、不能为 NULL。一张表一个主键,可由多列组成(复合主键)。SQLite 特性:声明 `INTEGER PRIMARY KEY` 的列会自动成为 **rowid 的别名**,自增--这是 SQLite 独有的便捷,`AUTOINCREMENT` 关键字其实很少需要(它会阻止 rowid 复用,语义更严格但性能略低)。

**NOT NULL**:列不能存 NULL。基础但重要,该加的列(如订单号、金额)一定加,否则后续查询和聚合会到处踩 NULL 的坑。

**UNIQUE**:列值(或列组合)不能重复。允许多个 NULL(NULL 在 SQLite 的 UNIQUE 里互不算重复,这是 SQL 标准的争议点,要留意)。用于用户名、车牌号等天然唯一的字段。

**CHECK**:自定义条件,存入时必须满足。如 `CHECK(年龄 >= 0)`、`CHECK(保费 >= 0)`。SQLite 会执行这个表达式,假则拒绝插入。注意 SQLite 的 CHECK 在历史上有"不强制"的隐患(某些版本对冲突容忍),最好配合应用层校验。

**DEFAULT**:默认值,插入时不给该列则用默认。如 `DEFAULT 0`、`DEFAULT CURRENT_TIMESTAMP`。常给状态、创建时间列。

**FOREIGN KEY(外键)**:引用另一表的主键,保证引用完整性。⚠️ **SQLite 的外键默认是关闭的**,必须每次连接后执行 `PRAGMA foreign_keys = ON;` 才生效--这是 SQLite 最隐蔽的坑,很多人建了外键以为生效了,实际没起作用。这是它和 MySQL/PG 的一大区别(后者默认就强制外键)。

约束可以写在列后(列级约束),也可单独成行(表级约束,复合主键/复合外键必须用表级)。工程建议:主键、NOT NULL、DEFAULT 在建表时就定好;UNIQUE/CHECK/外键按业务加,别怕约束多--约束是数据质量的保险,后期补数据远比加约束贵。

#### 串起来的一句话

DDL 搭骨架、DML 填血肉、DQL 取数据、约束定规则;记住 SELECT 的执行顺序(FROM→WHERE→GROUP BY→HAVING→SELECT→ORDER BY→LIMIT)、SQLite 的 `ALTER TABLE` 弱、`TRUNCATE` 没有、外键默认关闭这四个特异点,基础语法就够用了。

## 四、SQLite 专用命令(dot commands)
- `.open` / `.tables` / `.schema` / `.dump` / `.import` / `.mode` / `.headers`
- `sqlite3` CLI 命令行工具的使用


### SQLite 专用命令(dot commands)

`sqlite3` 是 SQLite 自带的命令行工具,在终端里直接操作数据库。它有两种工作模式:一种是输 SQL 语句(以分号结尾,所有数据库通用),另一种是输 **dot 命令**(以 `.` 开头、**不以分号结尾**,SQLite 专属)。dot 命令是 CLI 工具自己解释的,不是 SQL 标准,所以你在 Python 的 `sqlite3` 模块或别的客户端里用不了它们--它们只活在 `sqlite3` 这个命令行程序里。这是先要分清的:SQL 语句走分号、dot 命令走点号,前者是数据库语言,后者是工具快捷键。

#### 进入和打开数据库:`.open`

终端敲 `sqlite3` 回车,进入交互模式,这时还没绑定任何库;用 `.open 文件名` 打开(或创建)一个数据库文件。也可以一上来就 `sqlite3 test.db` 直接打开。`.open` 适合在交互里中途切换库,或者新建一个空库开始干活。退出用 `.exit` 或 `.quit`(或按 Ctrl+D)。

#### 看结构:`.tables` 和 `.schema`

`.tables` 列出当前库里所有表名,快速看有哪些表,等价于查 `sqlite_master` 但更省事。`.schema` 打印所有表的建表语句(`CREATE TABLE ...`),带表名参数 `.schema 表名` 只看某张表。这俩是摸清一个陌生数据库结构的起手式--拿到一个 `.db` 文件,先 `.tables` 看表,再 `.schema 表名` 看字段和约束,不用猜。`.schema` 还能看索引和视图的创建语句,因为它本质是输出 `sqlite_master` 里的 DDL。

#### 导出和导入:`.dump` 和 `.import`

`.dump` 把整个数据库导成**纯 SQL 文本**(一堆 `CREATE TABLE` + `INSERT`),重定向到文件就是一份可读、可重建的备份:`.dump > backup.sql`。这种备份叫"逻辑备份",跨平台、跨版本、可读、可改,是 SQLite 最常用的备份方式(物理备份则直接复制 `.db` 文件)。恢复时把 `backup.sql` 喂给 sqlite3 执行即可:`sqlite3 new.db < backup.sql`。

`.import` 用来把外部文件(通常是 CSV)灌进表里:`.import 文件名 表名`。它把文件每行作为一条记录插入。配合 `.mode csv` 和 `.headers on` 能正确处理逗号分隔和表头。注意 `.import` 默认按当前 `.mode` 的分隔符解析,所以**先设 mode 再 import**。这是把 Excel/CSV 数据快速入库的常用路径,比写脚本逐行 INSERT 快得多。

#### 控制输出格式:`.mode` 和 `.headers`

`.mode` 决定查询结果怎么显示,常用值:`list`(默认,竖线分隔)、`csv`(逗号分隔,导 CSV 用)、`column`(对齐成列,人看最舒服)、`insert`(输出成 INSERT 语句,导数据用)、`line`(每个字段一行,看宽表好用)、`html`(输出 HTML 表格)。`.headers on`(或 `.header on`)让结果第一行显示列名,默认是 off。这两个是日常调试最常调的--平时 `.mode column` + `.headers on` 看着最清楚,要导数据就切 `.mode csv` 或 `.mode insert`。另外 `.width` 能指定列宽(配合 column 模式),`.nullvalue 字符串` 能把 NULL 显示成指定字符(默认 NULL 不显示成空白,容易误以为是空字符串,设成 `.nullvalue <NULL>` 更清楚)。

#### 其他常用 dot 命令

`.help` 列出所有 dot 命令,记不住时随时查。`.databases` 列出当前挂载的数据库(含通过 `ATTACH` 挂进来的多个库)。`.indexes` 列索引,`.indexes 表名` 看某表的索引。`.read 文件名` 执行一个 SQL 脚本文件(把一堆 SQL 存成 `.sql` 批量跑)。`.show` 显示当前所有 dot 设置的值(排查"为什么输出格式不对"时有用)。`.save 文件名`(老版叫 `.backup`)把内存库或当前库另存。`.timer on` 打开查询耗时统计,调性能时看每条 SQL 跑多久。

#### 典型工作流

拿到一个 `.db` 文件,典型的 dot 命令工作流是:`sqlite3 文件.db` 进去 -> `.tables` 看有哪些表 -> `.schema 表名` 看结构 -> `.mode column` + `.headers on` 调好显示 -> 写 SELECT 查数据 -> 需要备份就 `.dump > bak.sql`。这套流程不依赖图形工具,纯终端就能摸清任何 SQLite 数据库,在服务器上没 GUI 时尤其有用。

#### 一个易混点

dot 命令和 SQL 是两套东西:dot 命令是 `sqlite3` CLI 工具的功能,SQL 才是数据库语言。所以你在 Python 里 `cur.execute(".tables")` 是**不会成功**的--Python 的 sqlite3 模块直接连数据库引擎,不经过 CLI 工具,没有 dot 命令这一层。想在程序里查表名,得查 `sqlite_master` 表:`SELECT name FROM sqlite_master WHERE type='table'`。记住"dot 命令只在 CLI 里有效",能避免很多困惑。

## 五、查询进阶
- 连接:inner / left / right / cross join
- 子查询:标量 / 列 / 表子查询 / 相关子查询 / EXISTS
- 集合运算:union / intersect / except
- 聚合、分组、排序、分页


### SQLite 查询进阶

单表查询会了之后,真实业务的数据散在多张表里,要把它们拼起来、嵌套起来、组合起来,靠的就是 join、子查询、集合运算这三套机制,外加聚合分组的进阶用法。这三套不是互斥的,实际查询里常嵌套混用。

#### 连接(JOIN):把多表按条件拼成宽表

JOIN 的本质是"按某个条件把两表的行配对,拼成更宽的行"。配对方式不同,就分出几种 JOIN。

**INNER JOIN(内连接)**:只保留两表都匹配上的行。A 和 B 的交集,匹配不上的行两边都丢。最常用,比如"订单 join 客户"只取有对应客户的订单。

**LEFT JOIN(左连接)**:保留左表所有行,右表匹配不上的字段填 NULL。查"所有客户及其订单(没下单的也要显示)"用它。左表是主体,右表是补充,补充不到就空着。这是业务里用得最多的 JOIN,因为大多数场景是"以一张主表为基准,去拼其他信息"。

**RIGHT JOIN(右连接)**:保留右表所有行,左表匹配不上填 NULL。SQLite **支持 RIGHT JOIN**(3.39+ 版本),但实践中几乎都把 RIGHT 改写成 LEFT(把表换边),因为 LEFT 更符合从左到右的阅读习惯,代码可读性更好。老版本 SQLite 不支持 RIGHT,养成用 LEFT 的习惯最稳。

**CROSS JOIN(交叉连接/笛卡尔积)**:不做任何匹配条件,左表每行和右表每行都配一遍,N×M 行。很少主动用(结果爆炸),通常是你**忘了写 ON 条件时意外产生的**--如果 join 结果行数远超预期,先查是不是漏了 ON。

**FULL OUTER JOIN(全外连接)**:左右两表都保留,匹配不上的都填 NULL。SQLite 从 3.39 才支持,之前要用 LEFT JOIN + UNION 模拟。用得少,知道有这回事即可。

JOIN 的关键点是 **ON 条件**写清楚配对规则(通常是 `A.id = B.a_id`),多表连时按顺序链式 join,每多一张表加一个 ON。性能上,join 的列最好有索引(尤其被 join 的外键列),否则是笛卡尔积再过滤,大数据量下很慢。一个易错点:LEFT JOIN 后在 WHERE 里过滤右表字段会把 NULL 行滤掉,等价于退化成 INNER JOIN--想在左表基准上筛右表条件,要把条件放进 ON 而不是 WHERE。

#### 子查询:查询里套查询

子查询是一条 SELECT 嵌在另一条 SQL 里,按"返回什么形状的结果"分类,不同形状能用的位置不同。

**标量子查询**:返回一行一列(单个值)。能用在任何需要单个值的地方,如 `WHERE 金额 > (SELECT AVG(金额) FROM 订单)`--比平均金额高的订单。标量子查询最安全,几乎不会出错,因为它就代表一个数。

**列子查询**:返回一列(多行一列)。常配合 `IN`/`NOT IN`/`ANY`/`ALL` 用,如 `WHERE 客户id IN (SELECT id FROM 黑名单)`。注意 `NOT IN` 遇到子查询里有 NULL 会出诡异结果(整个返回空),这是 SQL 的经典坑,改用 `NOT EXISTS` 更稳。

**表子查询**:返回多行多列,用在 FROM 后当一张临时表用,如 `SELECT * FROM (SELECT ... ) t WHERE ...`。必须给子查询起别名(`t`),否则报错。也叫"派生表"或"内联视图",用于先做一层聚合/过滤再查。

**相关子查询**:子查询里引用了外层查询的列,外层每查一行,子查询就执行一次。如"查比本部门平均薪资高的员工",子查询里的"本部门"依赖外层的当前行。逻辑直观但性能要警惕:外层 N 行就执行 N 次子查询,大数据量慢。很多相关子查询能改写成 JOIN,性能更好。

**EXISTS / NOT EXISTS**:相关子查询的常见形态,`WHERE EXISTS (SELECT 1 FROM B WHERE B.id = A.id)` 判断"是否存在匹配行",返回布尔。比 `IN` 更抗 NULL(`NOT EXISTS` 不会被 NULL 坑),大表上通常也比 `IN` 快,是判断"有没有关联记录"的首选写法。`SELECT 1` 是惯例,EXISTS 只关心有没有行返回,不关心列内容。

子查询的选择原则:要单个值用标量子查询;要判断存在性用 EXISTS;要一组值用 IN(但注意 NULL);要当临时表用表子查询。能用 JOIN 表达的优先 JOIN,通常更易读也更快,子查询留给 JOIN 表达不顺的场景(如相关判断、聚合后再过滤)。

#### 集合运算:把多个结果集纵向拼合

JOIN 是横向拼宽,集合运算是纵向拼高--把多个 SELECT 的结果按行上下叠起来。要求各 SELECT 的**列数相同、对应列类型兼容**(SQLite 比较宽松,不强求类型完全一致)。

**UNION**:两个结果集合并,**去重**(重复行只留一行)。`UNION ALL` 不去重,直接全保留。性能上 `UNION ALL` 更快(省了去重排序),**确定无重复或不在乎重复时优先用 ALL**。这是最常用的集合运算,比如把"本月订单"和"上月订单"叠成一个列表。

**INTERSECT**:取两个结果集的**交集**(都有的行才保留),去重。用得少,可被 `INNER JOIN` 或 `EXISTS` 替代。

**EXCEPT**:第一个结果集**减去**第二个(在 A 但不在 B 的行),去重。相当于"差集",也常被 `NOT EXISTS`/`LEFT JOIN ... IS NULL` 替代。

集合运算的坑:默认都去重,要保留重复加 ALL;排序要对最后一个 SELECT 用 ORDER BY(或把整体包进子查询再排),单给中间 SELECT 加 ORDER BY 无意义;列名以第一个 SELECT 为准。

#### 聚合、分组、排序、分页(进阶)

基础语法里讲了 GROUP BY/HAVING/ORDER BY/LIMIT 的骨架,这里补进阶要点。

**聚合函数**:`COUNT(*)` 数所有行(含 NULL)、`COUNT(列)` 数该列非 NULL 行数(差别常被忽略,出 bug)、`SUM/AVG` 忽略 NULL、`MIN/MAX` 忽略 NULL。`COUNT(DISTINCT 列)` 数去重后的个数。记住"聚合函数都忽略 NULL,除了 COUNT(*)",能解释很多统计结果对不上的问题。

**GROUP BY 进阶**:可按多列分组(先按 A 再按 B),粒度更细。SQLite 允许 SELECT 非分组列(别的数据库报错),但取组内任意行,不要依赖。分组后想筛组用 HAVING(能用聚合),不能用 WHERE(WHERE 在分组前、不能用聚合)。

**分组拼接(GROUP_CONCAT)**:SQLite 特色函数,把组内某列的值拼成一个字符串,如 `GROUP_CONCAT(客户名, ',')` 把同组的客户名拼成 "张三,李四"。做报表把多行折成一行时极好用,这是 SQLite 比 MySQL 的 GROUP_CONCAT 更灵活的地方(可指定分隔符和排序)。

**排序**:多列排序按声明优先级,`ORDER BY A, B DESC` 表示先按 A 升序,A 相同按 B 降序--DESC 只管它紧跟的那列,不会传染到 A。NULL 在排序里的位置:SQLite 里 NULL 默认排最前(升序时 NULL 在最上面),和 MySQL/PG 不同,处理 NULL 排序时用 `ORDER BY 列 IS NULL, 列` 之类显式控制。

**分页**:`LIMIT 每页数 OFFSET 跳过数`,第 2 页(每页10)就是 `LIMIT 10 OFFSET 10`。深分页(OFFSET 很大)性能差,因为要扫描跳过的行,大数据量改用"游标分页"(WHERE id > 上次最大id LIMIT n)。SQLite 的 LIMIT 语法和 MySQL 一致,跨库迁移无障碍。

#### 串起来

JOIN 横向拼宽、子查询纵向嵌套、集合运算纵向叠加,三者常组合:子查询先算个中间结果,FROM 后 JOIN 别的表,最后 UNION 拼上另一路结果。聚合分组穿插其中做统计。写复杂查询的思路是"先想清楚要什么样的最终表,再倒推每步用什么机制拼"--从结果倒推比从前往后试错高效得多。

## 六、索引
- 索引类型:单列 / 复合 / 唯一 / 部分(partial)/ 表达式索引
- 创建与删除
- 何时建、何时不建
- `EXPLAIN QUERY PLAN` 看执行计划


### SQLite 索引

索引是数据库为加速查询而维护的"额外数据结构"--本质是一棵 B-tree,把某列(或某些列)的值排好序存起来,查询时用二分查找快速定位,而不是全表扫描。代价是占额外空间、拖慢写入(每次增删改要同步更新索引)。理解索引的核心是记住一句话:**索引用空间换时间,加速读但拖慢写**。

#### 索引类型

**单列索引**:给一列建索引,最基本。`CREATE INDEX idx ON 表(列)`。适合该列常被 WHERE 过滤、ORDER BY 排序、JOIN 匹配。

**复合索引**:给多列组合建一个索引,`CREATE INDEX idx ON 表(列1, 列2)`。关键是**列顺序**--索引按声明顺序排序,先按列1、列1相同再按列2。所以能加速 `WHERE 列1=?`(最左前缀)、`WHERE 列1=? AND 列2=?`,但**加速不了** `WHERE 列2=?`(跳过了列1,索引用不上)。这叫"最左前缀原则":复合索引从最左列开始连续匹配才有用。设计复合索引时,把**选择性高(区分度大)的列放前面**、等值条件列在前范围条件列在后,这是通用优化原则。

**唯一索引**:索引列值不能重复,`CREATE UNIQUE INDEX idx ON 表(列)`。既加速查询又强制唯一性,等价于 UNIQUE 约束(UNIQUE 约束底层就是唯一索引)。主键自动就是个唯一索引。适合身份证号、订单号等天然唯一的字段。

**部分索引(partial index)**:只给满足条件的行建索引,`CREATE INDEX idx ON 表(列) WHERE 条件`。SQLite 是最早支持部分索引的数据库之一。价值是省空间、提效--比如只给"未完成订单"建索引,已完成的大量历史行不进索引,查询活跃数据时索引更小更快。常用于状态字段(只索引 status='active' 的行)。

**表达式索引**:给表达式或函数的结果建索引,`CREATE INDEX idx ON 表(表达式)`。比如经常按日期查 `WHERE date(时间戳)='2026-07-01'`,直接给"时间戳"建索引没用(因为查询用的是 `date()` 函数结果,不是原始列),这时 `CREATE INDEX idx ON 表(date(时间戳))` 才能命中。规则是:**WHERE 里对列套了函数,普通索引失效,得给函数结果建表达式索引**。这是很多人"建了索引却没用上"的常见原因。

#### 创建与删除

创建:`CREATE [UNIQUE] INDEX [IF NOT EXISTS] 索引名 ON 表(列 [ASC|DESC], ...) [WHERE 条件]`。索引名自定义(习惯 `idx_表_列`),`IF NOT EXISTS` 防重复创建报错。删除:`DROP INDEX [IF EXISTS] 索引名`。建索引时表会被短暂锁住(大表建索引要注意时机),SQLite 会在后台构建 B-tree。索引一旦建好,查询优化器会自动决定用不用,不用你在 SQL 里指定--这是声明式的好处,也是"建了索引为什么没生效"困惑的来源(用不用取决于优化器判断,不是建了就一定用)。

#### 何时建、何时不建

**该建**:经常出现在 WHERE 过滤条件里的列、JOIN 的连接列、ORDER BY/GROUP BY 的列、大表上(几千行以下的表建索引意义不大,全表扫描反而可能更快)。复合查询场景按最左前缀设计复合索引。

**不该建**:小表(数据量小全表扫更快)、写入极频繁读很少的列(索引拖慢写却换不来读收益)、选择性极低的列(如性别只有男/女,索引区分度低,优化器宁可直接扫表)、重复或冗余的索引(多个索引前缀重叠时,短的能覆盖长的,长的可删)。

判断原则是**读写权衡**:读多写少就多建,写多读少就少建;每个索引都要问"它加速的查询够频繁吗,值得付写入代价吗"。一个常见误区是见查询慢就加索引,结果索引越来越多、写入越来越慢、整体反而退化--索引该定期审视删掉不用的(`DROP INDEX` 不影响数据,只删加速结构,安全)。

#### EXPLAIN QUERY PLAN 看执行计划

这是验证"索引到底用没用"的唯一可靠方法。在 SQL 前加 `EXPLAIN QUERY PLAN`,SQLite 不执行查询,而是输出它**打算怎么查**:

```sql
EXPLAIN QUERY PLAN SELECT * FROM 订单 WHERE 客户id = 123;
```

输出里有关键字眼判断是否走索引:
- **`SEARCH 表名 USING INDEX 索引名`** -> 走了索引 ✅,快
- **`SEARCH 表名 USING PRIMARY KEY`** -> 走了主键索引 ✅
- **`SCAN 表名`** -> 全表扫描 ❌,没用上索引,大表会慢

看到 `SCAN` 就说明索引没生效,要查原因:是不是对列套了函数、是不是复合索引没满足最左前缀、是不是选择性太低优化器主动放弃索引、是不是用了 `OR` 导致部分条件用不上索引。EXPLAIN QUERY PLAN 是调优的核心工具,建完索引务必用它验证一遍,别凭感觉以为生效了。

#### 串起来

索引是用空间换时间的 B-tree,类型按需求选:单列管一个条件、复合管多条件(守最左前缀)、唯一管唯一性、部分管活跃子集、表达式管函数查询。建索引看读写权衡,删索引看是否冗余。建完一定用 `EXPLAIN QUERY PLAN` 验证是否真用了索引(看 SEARCH 还是 SCAN),这是从"我觉得快了"到"我知道为什么快"的分水岭。

## 七、事务与并发
- 事务:begin / commit / rollback
- ACID 特性在 SQLite 的实现
- 锁机制:共享锁 / 排他锁 / WAL 模式
- SQLite 的并发写限制(单写者)


### SQLite 事务与并发

事务是数据库保证"一组操作要么全做成、要么全不做"的机制,并发是多个连接同时操作同一数据库时的协调问题。SQLite 的事务模型和 MySQL/PostgreSQL 差异很大,根源还是它的"单文件、无服务"架构--所有并发协调都落在**文件锁**上,而不是内存里的锁管理器。理解这章要抓住一条主线:SQLite 是"单写者"模型,这是它并发能力的根本约束,所有机制都在围绕这个约束做权衡。

#### 事务:begin / commit / rollback

事务就是把多条 SQL 包成一个不可分割的单元。基本三件套:`BEGIN` 开启事务、`COMMIT` 提交(把改动落盘)、`ROLLBACK` 回滚(撤销自 BEGIN 以来的所有改动)。不显式写 BEGIN 时,SQLite 默认每条 SQL 自动包成一个事务(autocommit 模式),所以你批量插 1000 条如果不手动 BEGIN,实际是 1000 个独立事务,每条都单独落盘一次,极慢--这是 SQLite "插入慢"的头号原因,套一个 `BEGIN ... COMMIT` 能快几十倍。

事务的典型用法:`BEGIN; INSERT ...; INSERT ...; COMMIT;`,中间任何一步出错就 `ROLLBACK` 整体撤销。在 Python 的 `sqlite3` 模块里,用 `with conn:` 上下文管理器会自动提交或回滚(正常退出提交、异常退出回滚),比手动管理更稳。SQLite 还支持 SAVEPOINT(嵌套事务),在事务里设保存点,可以部分回滚而不撤销整个事务,复杂流程时有用。

#### ACID 在 SQLite 的实现

ACID 是事务的四个保证,SQLite 都支持,但实现方式有自己的特点:

**A(原子性)**:靠事务日志(rollback journal 或 WAL)实现,提交前先把原始数据写日志,出问题靠日志恢复,保证要么全做要么全不做。

**C(一致性)**:由原子性 + 约束(主键、外键、CHECK 等)共同保证,事务提交后数据库仍满足所有约束。注意外键默认关闭(前面提过),开着才真正参与一致性保证。

**I(隔离性)**:SQLite 默认隔离级别是 **SERIALIZABLE(串行化)**,这是最强的隔离--读到的都是已提交的、且不受并发写影响。听起来很美,代价是并发性能被牺牲(单写者)。SQLite 不像 MySQL 有多级隔离可选,它基本就一个模式。

**D(持久性)**:提交后数据落盘不丢。但 SQLite 有个微妙点:默认的 journal 模式下,COMMIT 返回成功只保证写到了 OS 缓存,**不保证真正刷到磁盘**,极端断电仍可能丢最近提交。要更强持久性得 `PRAGMA synchronous=FULL`(每次提交都 fsync 强制刷盘,慢但最安全)。这是"性能 vs 安全"的权衡,默认值偏性能。

#### 锁机制:共享锁 / 排他锁 / WAL

SQLite 的锁是**文件级**的(锁整个数据库文件,不是行级),这决定了它的并发粒度粗。默认 rollback journal 模式下有五种锁状态,简化理解为两级:

**共享锁(SHARED,读锁)**:多个连接可同时持有,用于读。读读不互斥。

**排他锁(EXCLUSIVE,写锁)**:写操作要先拿 RESERVED 再升级到 EXCLUSIVE,持有期间其他连接既不能写也不能读(默认模式下写会阻塞读)。这是默认模式下"写会卡住读"的根源。

默认模式(rollback journal)的并发表现:多个读可以并发;但一旦有写,写会独占文件,其他读和写都得等。写多场景下性能差。

**WAL 模式(Write-Ahead Logging)**是 SQLite 并发的关键升级,`PRAGMA journal_mode=WAL` 开启。WAL 把写操作先追加到独立的 WAL 文件,不直接改主库文件,于是**读和写不再互斥**--读连接读主库快照,写连接往 WAL 追加,各自不挡对方。这大幅改善了"读写并发"的场景(一个写、多个读同时进行)。但注意:**WAL 没有突破单写者限制**,同时仍只能有一个写连接,多个写还是排队。WAL 的额外好处是写入更快(顺序追加比随机改快)、崩溃恢复更快。缺点是多了 WAL 和 SHM 文件、不支持网络文件系统(WAL 不能放 NFS 上)。生产里读写并发的 SQLite 基本都开 WAL。

#### SQLite 的并发写限制(单写者)

这是 SQLite 最硬的约束:**整个数据库同一时刻只能有一个写事务**。多个连接想写,得排队拿写锁,拿不到的就 `SQLITE_BUSY` 报错或阻塞。这不是 bug 是设计--文件级锁没法做行级并发写,要突破就得换成有服务进程的 MySQL/PG。

应对方式有几种:一是开 WAL(改善读写并发,但写仍单线程);二是设 **busy_timeout**(`PRAGMA busy_timeout=5000`),让拿不到锁的连接等 5 秒再报错,而不是立刻失败,这是最实用的配置;三是应用层加自己的锁或写队列,串行化写操作;四是把写频率降下来(批量写、合并事务)。如果业务写并发很高(比如每秒几十上百写),SQLite 基本就到极限了,该上 MySQL/PG。

判断法则:读多写少 -> SQLite + WAL 完全够用;写并发高 -> 换数据库。这也是为什么 SQLite 适合嵌入式、单机工具、中小站点,不适合高并发写入的生产中央库--根源都在"单写者"。

#### 串起来

事务保证一组操作的原子性,SQLite 默认每条 SQL 一个事务,批量操作要手动 BEGIN 才快;ACID 全支持但隔离是串行化、持久性默认偏性能(synchronous 调);并发靠文件级锁,默认模式写阻塞读,开 WAL 后读写不互斥但写仍单线程;单写者是根本约束,读多写少用 SQLite+WAL,写并发高就该换 MySQL/PG。所有并发特性的根源都是"单文件 + 无服务进程",这是架构决定的天花板,优化只能逼近它不能突破它。

## 八、内置函数与表达式
- 文本 / 数值 / 聚合 / 日期时间函数
- NULL 的处理(coalesce / ifnull / nullif)
- CASE 表达式

### SQLite 内置函数与表达式

SQLite 内置一批函数,按用途分文本、数值、聚合、日期时间几大类,加上 NULL 处理函数和 CASE 表达式,基本覆盖日常数据处理。这些函数是 SQL 标准的子集,大部分能跨 MySQL/PostgreSQL 通用,但 SQLite 有几个自己的特色函数(尤其日期时间)值得单独留意。

#### 文本函数

最常用的几个:`LENGTH(s)` 返回字符数;`UPPER(s)`/`LOWER(s)` 转大小写;`TRIM(s)`/`LTRIM`/`RTRIM` 去两端/左/右空格;`SUBSTR(s, 起, 长度)` 截子串(下标从 1 开始,不是 0);`REPLACE(s, 旧, 新)` 替换;`INSTR(s, 子串)` 返回子串首次出现位置,找不到返回 0(不是 -1,和 Python 不同)。拼接字符串 SQLite 用 `||` 运算符(`'a' || 'b'` 得 `'ab'`),也支持 `CONCAT`(新版),但 `||` 更通用更稳。`PRINTF(fmt, ...)` 按 C 语言格式化字符串,做"补零""保留小数"很方便,如 `PRINTF('%05d', 42)` 得 `'00042'`。`GROUP_CONCAT` 严格说是聚合函数(在聚合那块讲),但常和文本一起记,把多行值拼成一个串。

#### 数值函数

`ABS(n)` 绝对值;`ROUND(n, 位数)` 四舍五入(位数可负,`ROUND(123.45, -1)` 得 120);`CAST(值 AS 类型)` 类型转换(如 `CAST('123' AS INTEGER)`),注意 CAST 在 SQLite 里受类型亲和影响,转不动时不报错而是给个怪结果,要小心;`MAX`/`MIN` 既是聚合函数也能当标量函数用(`MAX(1,2)` 返回 2,标量用法取多个参数的最大值,这和别的数据库不同,SQLite 的 MAX/MIN 一函数两用法)。`RANDOM()` 返回随机整数,`RANDOMBLOB(n)` 生成 n 字节随机二进制。注意 SQLite 没有 `CEIL`/`FLOOR`(要用 `CAST` 配合运算模拟),也没有 `POWER` 直接函数(用 `||` 不行,得靠表达式或 `POW` 看版本),数值函数比 MySQL/PG 少,复杂运算常要在应用层做。

#### 聚合函数

聚合函数把多行聚合成一个值,配合 GROUP BY 用:`COUNT(*)` 数所有行(含 NULL 行)、`COUNT(列)` 数该列非 NULL 值的行数(两者区别常被忽略,出 bug);`SUM(列)` 求和(忽略 NULL);`AVG(列)` 平均(忽略 NULL,注意是"非 NULL 值的平均",分母不含 NULL 行);`MIN`/`MAX` 最小最大值(忽略 NULL);`GROUP_CONCAT(列, 分隔符)` 把组内值拼成串(分隔符默认逗号),SQLite 特色,做报表把多行折一行极好用;`COUNT(DISTINCT 列)` 数去重后的个数。核心要点还是那句:**所有聚合函数都忽略 NULL,除 COUNT(\*)**。理解这条能解释 90% 的"统计数字对不上"问题。

#### 日期时间函数

这是 SQLite **和别的数据库差异最大**的一块。SQLite 没有 `DATE`/`DATETIME` 类型(前面讲过,实际存为 TEXT 或 INTEGER),日期处理全靠几个函数,且时间用**字符串或修饰符**表示,不是对象。核心函数:

- `DATE(时间, 修饰符...)` 返回日期字符串 `'YYYY-MM-DD'`
- `DATETIME(时间, 修饰符...)` 返回 `'YYYY-MM-DD HH:MM:SS'`
- `TIME(...)` 返回时间部分
- `STRFTIME(格式, 时间)` 按格式串输出,最灵活,如 `STRFTIME('%Y年%m月', 'now')` 得 `'2026年07月'`

时间可以用 `'now'`(当前 UTC 时间)、`'2026-07-01'`、`'2026-07-01 10:30:00'` 等字符串,也可用 Julian day 数值。修饰符做加减:`DATE('now', '+1 day')` 明天、`DATE('now', '-1 month')` 上月、`DATE('now', '+1 year')` 明年、`DATETIME('now', 'localtime')` 转本地时区。求差用 `JULIANDAY`:`(JULIANDAY('2026-07-01') - JULIANDAY('2026-06-01'))` 得天数差。

要点:SQLite 的时间函数都基于 **UTC**,`'now'` 是 UTC 当前时间,要本地时间加 `'localtime'` 修饰符;时间字符串格式严格(必须 `YYYY-MM-DD` 或带 `T`/空格的完整形式),格式不对函数返回 NULL 不报错;时间列存 TEXT 时用 `YYYY-MM-DD HH:MM:SS` 格式才能直接字符串比较排序(字典序就是时间序,这是个实用技巧)。相比 MySQL 的 `NOW()`、`DATE_ADD` 等,SQLite 的日期函数更"原始",但灵活,做精算按月/按季度统计时 `STRFTIME('%Y-%m', 日期)` 分组是常用套路。

#### NULL 的处理(coalesce / ifnull / nullif)

NULL 在 SQL 里特殊--它"不等于任何值,包括它自己",任何运算碰到 NULL 结果都是 NULL(NULL 传染性),这让处理 NULL 成高频需求。SQLite 提供三个函数:

- **`COALESCE(值1, 值2, ...)`**:返回第一个非 NULL 的值,全为 NULL 才返回 NULL。最通用,可接多个参数。`COALESCE(折扣, 0)` 把 NULL 折扣当 0,是标准写法。
- **`IFNULL(值, 默认)`**:和 COALESCE 类似但**只接两个参数**,是 SQLite/MySQL 特有(SQL 标准是 COALESCE)。`IFNULL(折扣, 0)` 等价于 `COALESCE(折扣, 0)`。两个参数时用 IFNULL 更简洁,多参数必须用 COALESCE。
- **`NULLIF(值1, 值2)`**:两值相等返回 NULL,否则返回值1。用途是把特定值"变成 NULL",常用来避免除零:`NULLIF(分母, 0)` 在分母为 0 时返回 NULL,让除法结果为 NULL 而不是报错或得 inf。也常用于"把 0 当作缺失值"的场景。

补充一个运算符:**`IS NULL` / `IS NOT NULL`** 判断是否为 NULL(不能用 `= NULL`,因为 NULL 不等于任何值,`= NULL` 永远返回 NULL 不是真)。SQLite 还支持 `IS`/`IS NOT` 做 NULL 安全的等值比较(`a IS b` 在 a 和 b 都是 NULL 时为真,而 `a = b` 为 NULL),这在比较可能含 NULL 的列时很有用,是 SQLite 的便利扩展。

#### CASE 表达式

CASE 是 SQL 里的"if-else",两种写法:

**简单 CASE**:`CASE 列 WHEN 值1 THEN 结果1 WHEN 值2 THEN 结果2 ELSE 默认 END`,按列值匹配。

**搜索 CASE**(更灵活,推荐):`CASE WHEN 条件1 THEN 结果1 WHEN 条件2 THEN 结果2 ELSE 默认 END`,条件可以是任意布尔表达式,不限于相等比较。

CASE 返回第一个满足条件的 THEN 值,都不满足返回 ELSE(没 ELSE 则返回 NULL)。用途广泛:做分类标签(`CASE WHEN 赔付率>0.7 THEN '高风险' WHEN >0.5 THEN '中' ELSE '低' END`)、做条件聚合(`SUM(CASE WHEN 状态='已完成' THEN 金额 ELSE 0 END)` 算已完成订单总金额,这是把行级条件塞进聚合的经典技巧)、做数据脱敏、做报表列转。CASE 是 SQL 里表达分支逻辑的唯一方式,熟练它能省掉很多"查出数据再在应用层 if-else"的往返。注意 CASE 是表达式不是语句,可以出现在 SELECT、WHERE、ORDER BY 等任何能用表达式的地方,不是只能 SELECT 里。

#### 串起来

文本/数值/聚合函数大部分跨库通用,记 `||` 拼接、`SUBSTR` 下标从 1、`PRINTF` 格式化这几个 SQLite 特色即可;日期时间函数是 SQLite 最独特的一块,全靠字符串+修饰符,`'now'` 是 UTC 要加 `'localtime'`,按 `STRFTIME('%Y-%m', 日期)` 分组是统计利器;NULL 处理三件套 COALESCE/IFNULL/NULLIF 应对 NULL 传染,`NULLIF(分母,0)` 防除零是标配;CASE 是 SQL 的 if-else,条件聚合 `SUM(CASE WHEN...)` 是高频技巧。这些函数表达式混在一起,就是日常数据加工的全部工具箱。

- view(视图)
- trigger(触发器)

### SQLite 视图与触发器

视图和触发器是两种"把逻辑下沉到数据库"的机制--视图把查询逻辑固化成虚拟表,触发器把动作逻辑挂在表的事件上自动执行。SQLite 对这两者的支持都比较完整,但有一个重要缺失:**没有存储过程**。这三者的关系理解清楚,就知道 SQLite 能把多少业务逻辑放进数据库、又有哪些得放回应用层。

#### 视图(view)

视图是一条 SELECT 语句保存下来的"虚拟表"--它不存数据,只存定义,每次查视图时动态执行底层 SQL。创建用 `CREATE VIEW 视图名 AS SELECT ...`,删除用 `DROP VIEW 视图名`。查视图和查普通表语法完全一样(`SELECT * FROM 视图名`),对使用者透明。

视图的价值有三:一是**简化复杂查询**,把多表 join、子查询、聚合封装成一个视图,业务侧只查视图不用重写那一坨 SQL;二是**提供逻辑抽象**,底层表结构变了,只要视图定义调整,上层查询不受影响,相当于给应用一个稳定的数据接口;三是**做权限隔离**,虽然 SQLite 没有用户权限系统,但概念上可以只暴露视图不暴露原表,隐藏敏感列。SQLite 的视图默认是**只读**的(不能直接 INSERT/UPDATE/DELETE),想通过视图改数据得建 `INSTEAD OF` 触发器(见下)。视图不占存储(只存定义几行 SQL),但每次查都要重算底层查询,所以视图嵌套太深或底层查询太重,性能会成问题--视图是"便利"不是"加速",它本身不会让查询变快。

#### 触发器(trigger)

触发器是挂在表上的"自动动作"--当表发生 INSERT/UPDATE/DELETE 时,数据库自动执行一段预先定义的逻辑。创建用 `CREATE TRIGGER 触发器名 {BEFORE|AFTER|INSTEAD OF} {INSERT|UPDATE|DELETE} ON 表名 FOR EACH ROW BEGIN ... END`,删除用 `DROP TRIGGER`。

几个要点:触发器可指定 `BEFORE`(事件发生前执行,常用于校验或修改待写入的值)或 `AFTER`(事件发生后执行,常用于联动更新其他表或记日志);`FOR EACH ROW` 是 SQLite 唯一支持的模式(行级触发器,每个受影响行都触发一次,没有语句级触发器);触发器体内能通过 `NEW` 和 `OLD` 引用改动的行数据--INSERT 时只有 `NEW`(新值)、DELETE 时只有 `OLD`(旧值)、UPDATE 时两者都有(`OLD` 是改前、`NEW` 是改后)。

典型用途:数据校验(BEFORE INSERT 检查业务规则,不满足就抛 RAISE 阻止)、联动更新(AFTER INSERT 订单时自动更新客户的最近下单时间)、审计日志(AFTER UPDATE 把改动记到日志表)、派生字段维护(主表改了自动同步汇总表)。还有一个 SQLite 特定用法:**`INSTEAD OF` 触发器让视图可写**--对视图做 INSERT 时,原本不行,但建一个 `INSTEAD OF INSERT ON 视图` 的触发器,把插入拆解到底层多张表,视图就"可写"了,这是处理多表视图更新的标准套路。

触发器的代价是**逻辑隐蔽**:业务逻辑藏在数据库里,应用代码看不见,调试和排错时容易忘了"还有个触发器在偷偷干活"。所以触发器要克制使用,只放那些"必须由数据库保证、绕不过去"的逻辑(如审计、强一致联动),业务规则尽量放应用层更透明。SQLite 的触发器功能相对完整,但不能在触发器里做复杂事务控制,能力比 PostgreSQL 弱。

#### SQLite 没有存储过程

这是 SQLite 和 MySQL/PostgreSQL 的重要区别:**SQLite 不支持存储过程(procedure)、也没有自定义函数(UDF)的 SQL 语法**。你没法像 MySQL 那样 `CREATE PROCEDURE 名(参数) BEGIN ... END` 把一段 SQL 逻辑存进数据库反复调用。

原因还是架构:存储过程需要数据库服务进程来执行和缓存逻辑,SQLite 是嵌入式库,跑在应用进程里,没有独立的"数据库服务"来承载存储过程。SQLite 的设计哲学是**逻辑放应用层、数据库只管存和查**,这和它"轻量嵌入式"的定位一致。

实际影响:原本想用存储过程封装的业务逻辑,在 SQLite 里要么写成应用层的 Python 函数(调用多条 SQL),要么用视图(只读查询)或触发器(事件联动)部分替代。复杂的数据处理流程基本都得回到应用层用代码组织,这反而更符合现代开发实践--存储过程把业务逻辑锁死在数据库里,可移植性和可维护性都不好,SQLite 不支持它,某种程度上倒逼你把逻辑放对地方。需要"重复执行某段 SQL"时,用应用层的函数封装 + 参数化查询,效果等同且更灵活。

#### 串起来

视图把查询逻辑固化成虚拟表,只读(除非配 INSTEAD OF 触发器),用于简化、抽象、隔离;触发器把动作逻辑挂到表事件上自动跑,用 NEW/OLD 引用改动行,用于校验、联动、审计,但要克制以防逻辑隐蔽;SQLite 没有存储过程,复杂业务逻辑放应用层用代码组织,这既是架构限制也是现代实践推荐。三者合起来,SQLite 能把"查询封装"和"事件联动"放进数据库,但"流程封装"不行--逻辑该放哪层,SQLite 的设计已经替你做了选择:重逻辑回应用层。

### SQLite Python 接口

Python 标准库自带 `sqlite3` 模块,无需安装任何东西,`import sqlite3` 就能用。这是 SQLite 和 Python 配合的最大优势--零依赖、开箱即用,特别适合单机工具、脚本、数据分析。整个接口围绕"连接-游标-执行"三件套展开,理解了这个骨架,加上参数化查询、事务控制、上下文管理,就掌握了用 Python 操作 SQLite 的全部日常能力。

#### sqlite3 模块与连接/游标/执行

核心对象有两个:**Connection(连接)**和 **Cursor(游标)**。Connection 代表和数据库的会话,`conn = sqlite3.connect('test.db')` 打开(文件不存在自动创建);传 `':memory:'` 则建在内存里(程序退出即消失,适合测试)。Cursor 是执行 SQL 和遍历结果的句柄,`cur = conn.cursor()` 创建。执行 SQL 用 `cur.execute(sql, 参数)` 或批量 `cur.executemany(sql, 参数列表)`,查完用 `cur.fetchone()`(取一行)/`cur.fetchall()`(取全部)/`cur.fetchmany(n)`(取 n 行)读结果。

一个简化认识:Connection 也直接支持 `conn.execute(sql)`,会隐式创建游标并返回它,简单场景可以省去显式 cursor。但严谨写法仍建议用 cursor,尤其一个连接要交错执行多条语句时,共用 cursor 会让结果集互相干扰。执行完**修改类 SQL(INSERT/UPDATE/DELETE)要 `conn.commit()` 才真正落盘**,不 commit 的话改动在连接关闭时就丢了(或回滚);查询类则不需要 commit。最后 `conn.close()` 关闭连接释放资源。

#### 参数化查询(防注入)

这是最关键的安全实践。**永远不要用字符串拼接或 f-string 拼 SQL**,比如 `f"SELECT * FROM 用户 WHERE 名字='{name}'"`--当 name 是 `' OR '1'='1` 时,整条 SQL 被篡改成无条件真,这就是 SQL 注入,能删库能泄密。正确做法是**参数化查询**:SQL 里用占位符,参数单独传,让 sqlite3 模块负责安全转义。

SQLite 的占位符有两种:`?`(问号风格,最常用)和 `:名字`/`%s`(命名风格)。示例:

```python
cur.execute("SELECT * FROM 用户 WHERE 名字 = ?", (name,))          # 问号风格,参数是元组
cur.execute("SELECT * FROM 用户 WHERE 名字 = :n", {"n": name})     # 命名风格,参数是字典
cur.executemany("INSERT INTO t VALUES(?,?)", [(1,'a'),(2,'b')])     # 批量
```

参数化不只防注入,还更快(同一条 SQL 可被预编译复用)更稳(自动处理引号、特殊字符、NULL)。记住一条铁律:**只要 SQL 里有来自外部的值,就用占位符,绝不拼接**。元组单参数要写成 `(name,)` 不是 `(name)`(后者是表达式不是元组),这是新手常犯的语法坑。

#### 事务控制方式

Python 的 sqlite3 模块默认是 **autocommit off** 模式(老版本 Python)或接近自动事务(新版行为有调整),实际表现:执行修改类 SQL 后,需要 `conn.commit()` 提交,或 `conn.rollback()` 回滚。批量写不 commit 的话,改动只在当前连接可见,关连接即丢。

关键认知:**sqlite3 模块在 autocommit 行为上有过历史变化**,Python 3.6 之前默认自动提交,之后默认需要手动 commit,行为略绕。最稳的写法是**显式控制**:要么手动 `commit()`/`rollback()`,要么用下面讲的 `with conn:` 上下文管理。另外,DDL 语句(CREATE/DROP)在 sqlite3 里会**自动提交**当前挂起的事务(因为 SQLite 引擎本身对 DDL 这么处理),这是个容易困惑的点--执行 DDL 后之前的未提交改动会被顺带提交。

 isolation_level 参数能调事务行为:`sqlite3.connect('db', isolation_level=None)` 关掉模块层的自动事务管理,进入纯 autocommit 模式(每条 SQL 立即生效),这时要自己用 `BEGIN` 显式开事务。高级用法才碰,日常用默认 + 显式 commit 即可。

#### 上下文管理(with)

`with conn:` 是管理事务的推荐方式,它自动处理提交和回滚:**with 块正常结束自动 commit,抛异常自动 rollback**。这让事务代码既简洁又安全,不用手写 try/except/commit/rollback:

```python
with conn:
    conn.execute("INSERT INTO t VALUES(?,?)", (1,'a'))
    conn.execute("INSERT INTO t VALUES(?,?)", (2,'b'))
# 正常结束自动 commit;中间抛异常自动 rollback
```

注意一个细节:**`with conn:` 管的是事务,不是连接关闭**。with 结束后连接还开着,能继续用,要关连接仍需 `conn.close()`(或用 `contextlib.closing`)。这和 `with open(...) as f:` 不同,后者关的是文件。容易混淆,记住"with 连接管事务、不管关闭"。

游标也能 `with cur:`,但用得少。日常模式:连接用 with 管事务、程序结束时 close 连接;或者用 `contextlib.closing(sqlite3.connect(...))` 让 with 同时管关闭。工程上推荐把数据库操作封装成函数/类,连接生命周期统一管理,避免到处 open/close。

#### 与 ORM(SQLAlchemy)的关系

直接用 `sqlite3` 模块叫"原生 SQL"或"裸驱动"--你手写 SQL、手动处理结果到对象的映射。当项目变大、表多了、对象关系复杂时,手写 SQL 和手工映射很繁琐,ORM(对象关系映射)就登场了。Python 主流 ORM 是 **SQLAlchemy**,它把表映射成 Python 类、行映射成对象、SQL 操作映射成方法调用,让你用 Python 代码而非 SQL 操作数据库。

关系是分层:SQLAlchemy 底层仍然通过驱动(如 `sqlite3` 或更底层的 pysqlite)连数据库,它只是在上层封装。`sqlite3` 是"直接和 SQLite 引擎对话",SQLAlchemy 是"用 Python 对象表达数据操作,再自动翻译成 SQL"。所以 SQLAlchemy 不是替代 sqlite3,而是在它之上加抽象层。

用哪个看场景:**小工具、脚本、数据分析、想精细控制 SQL** -> 直接 `sqlite3`,轻量、透明、性能好、无额外依赖;**中大型项目、表关系复杂、团队协作、想跨数据库可移植** -> SQLAlchemy,开发效率高、防注入自动处理、能换底层数据库(SQLite 换 PostgreSQL 改个连接串就行)。还有个折中:SQLAlchemy Core(只做 SQL 表达式封装,不做对象映射),介于裸 SQL 和完整 ORM 之间,保留了 SQL 的可控又省去手工拼字符串。

一个实战路径:学 SQLite 阶段先用 `sqlite3` 原生接口,把 SQL 语法和事务搞透;项目复杂了再上 SQLAlchemy。**不建议一上来就 ORM**--没写过原生 SQL 的人用 ORM,出问题时既看不懂 ORM 生成的 SQL、也不知道怎么调,基础不牢。先裸 SQL 再 ORM 是更稳的成长路径,这也契合 SQLite "逻辑放应用层、用代码组织"的哲学。

#### 串起来

`sqlite3` 模块零依赖开箱即用,核心是 Connection 管会话、Cursor 执行查询、修改类要 commit;参数化查询(占位符 `?`/`:名`)是防注入的铁律,绝不拼接 SQL;事务用 `with conn:` 自动提交回滚最稳,但 with 不管连接关闭;复杂项目可上 SQLAlchemy ORM,但建议先吃透原生 SQL 再用 ORM,基础才牢。整个接口设计简洁,把 SQL 能力完整暴露给 Python,加上 SQLite 本身的零配置,Python+SQLite 是单机数据处理的黄金组合。

## 十二、性能优化
- `EXPLAIN QUERY PLAN` 分析查询
- 索引优化策略
- `PRAGMA` 调优(缓存、同步模式等)
- `ANALYZE` 统计信息收集


### SQLite 性能优化

SQLite 的性能优化思路和大型数据库一致,但有几个 SQLite 特有的旋钮(PRAGMA)和约束。整体顺序是:**先定位慢在哪,再针对性优化**--不要盲目加索引或调参数,优化没有银弹,一切以测量为前提。核心工具是 EXPLAIN QUERY PLAN 看执行计划、ANALYZE 给优化器喂统计、索引优化加速查询、PRAGMA 调底层行为,四者配合覆盖绝大多数优化场景。

#### EXPLAIN QUERY PLAN 分析查询

这是优化的第一步也是唯一可靠的诊断工具。在 SQL 前加 `EXPLAIN QUERY PLAN`,SQLite 输出它**打算怎么执行**这条查询,而不是真执行。关键看输出里的动词:

- **`SEARCH 表名 USING INDEX 索引名`** 或 `USING PRIMARY KEY` -> 走索引 ✅,快
- **`SCAN 表名`** -> 全表扫描 ❌,没用上索引,大表会慢
- **`SEARCH 表名 USING COVERING INDEX`** -> 覆盖索引 ✅✅,连回表都省了,最快(索引里就有所需全部列)

看到 SCAN 就要查为什么没走索引:是不是对列套了函数(普通索引失效)、是不是复合索引没满足最左前缀、是不是 `OR` 条件导致部分用不上、是不是选择性太低优化器主动放弃。EXPLAIN QUERY PLAN 还会暴露"临时 B-tree"(排序或分组时产生,大表上慢,可加对应索引消除)和"自动覆盖索引"(新版 SQLite 会自动建临时索引,但仍不如显式索引稳)。养成习惯:**写完一条慢查询,先 EXPLAIN 一遍再谈优化**,从"我觉得快了"变成"我知道为什么快"。

#### 索引优化策略

索引是 SQLite 性能优化的主战场,前面"索引"那块讲了类型和何时建,这里补优化策略。第一,**按查询模式建复合索引而非多个单列索引**:一条 `WHERE a=? AND b=?` 用 `(a,b)` 复合索引一次命中,比 a、b 各一个单列索引更高效(优化器通常只用其中一个)。第二,**覆盖索引**:把 SELECT 需要的列都塞进复合索引,查询在索引里就能拿全数据,不用回表查主数据,这是把查询压到最快的关键技巧,`CREATE INDEX idx ON 表(a, b, c)` 配 `SELECT b,c FROM 表 WHERE a=?` 即覆盖。第三,**最左前缀**:复合索引按声明顺序匹配,设计时把等值条件列放前、范围条件列放后、高区分度列放前。第四,**警惕索引失效场景**:对列套函数(`WHERE date(列)`)走不了普通索引,改用表达式索引;`LIKE 'xxx%'` 能用索引但 `LIKE '%xxx'`(前缀通配)用不了;`OR` 两边都得有索引否则全表扫。第五,**定期清理冗余索引**:`(a,b)` 已存在时 `(a)` 就是冗余的(前者覆盖后者),多余的索引只拖慢写入不带来收益,该删就删。验证手段永远是 EXPLAIN QUERY PLAN,建完索引看是不是真的 SEARCH USING 了。

#### PRAGMA 调优

PRAGMA 是 SQLite 特有的配置命令,调的是数据库引擎的底层行为,不在 SQL 标准里。优化相关的几个关键 PRAGMA:

**`PRAGMA journal_mode=WAL`**:前面事务那章讲过,开启 WAL 后读写不互斥、写入更快。这是 SQLite 性能优化的第一刀,几乎所有读写并发的场景都该开。开了 WAL 后多一个 `-wal` 和 `-shm` 文件,正常。

**`PRAGMA synchronous`**:控制持久性 vs 性能权衡。默认 `FULL`(每次提交 fsync 强制刷盘,最安全最慢);`NORMAL`(WAL 模式下安全且快很多,推荐生产用);`OFF`(不保证刷盘,最快但断电可能丢数据,仅测试用)。开 WAL 后配 `synchronous=NORMAL` 是经典组合,安全性和性能平衡得最好。

**`PRAGMA cache_size`**:调整 SQLite 的内存页缓存大小,默认可能偏小,大内存机器调大(如 `-200000` 表示约 200MB)能减少磁盘 IO,读密集场景明显加速。正负号表示页数 vs KB,按需设。

**`PRAGMA temp_store=MEMORY`**:临时表和中间结果放内存而非磁盘,排序/分组重的查询受益。

**`PRAGMA mmap_size`**:内存映射文件 IO,大数据库读取能提速,但要注意它和 WAL 的兼容性及 32 位系统限制。

**`PRAGMA page_size`** 和 `**PRAGMA encoding`**:建库时设定,page_size 大(如 4096/8192)对大库更高效,但**只能建库时设,已有数据改不了**(要改得 VACUUM 重建)。这些"一次性"PRAGMA 建库时就要想好。

PRAGMA 调优原则:**WAL + synchronous=NORMAL 是通用起点**,再按场景调 cache_size/temp_store,每次只调一个看效果,别一把全开。PRAGMA 是会话级的(每次连接要重新设),生产里通常在连接初始化代码里统一设。

#### ANALYZE 统计信息收集

SQLite 的查询优化器决定"用不用索引、用哪个索引、join 顺序"时,依赖**统计信息**--各表行数、各列值分布、索引选择性等。没有统计信息时优化器只能瞎猜(默认假设列均匀分布),可能选错索引或 join 顺序,导致慢查询。`ANALYZE` 命令扫描数据库收集这些统计,存进 `sqlite_stat1` 等系统表,优化器据此做更优决策。

用法:`ANALYZE`(全库)、`ANALYZE 表名`(单表)、`ANALYZE 索引名`(单索引)。执行后优化器有了真实分布数据,复杂查询(多表 join、多索引可选)的执行计划常显著改善。ANALYZE 本身要扫全库,大库上耗时,所以不用频繁跑--数据变化大时(大批量插入后、数据分布显著改变后)跑一次即可。新版 SQLite 还支持**自动 ANALYZE**(`PRAGMA automatic_index` 相关,以及默认开启的 `sqlite_stat1` 自动维护),但复杂库仍建议手动 ANALYZE 一次确保统计新鲜。判断要不要 ANALYZE:用 EXPLAIN QUERY PLAN 看执行计划不合理(明明有更好索引却选了差的),且统计可能过时,就跑一次 ANALYZE 再看。

#### 串起来

优化流程是:EXPLAIN QUERY PLAN 定位(SCAN 还是 SEARCH、有没有临时 B-tree)-> 索引优化(复合索引、覆盖索引、清冗余、避开失效场景)-> PRAGMA 调底层(WAL + synchronous=NORMAL 起步,再调缓存)-> ANALYZE 喂统计(数据分布变了就跑,让优化器选对索引)。核心心法:**测量优先、一次只改一个变量、验证后再固化**。SQLite 优化没有黑魔法,WAL+合理索引+正确统计这三件事做好,绝大多数场景性能就够,真正的瓶颈往往在"没用索引"或"没用事务批量写"这两个基础问题上,而不是 PRAGMA 没调对。

### SQLite 进阶特性

这些是 SQLite 较新版本陆续加入的高级能力,日常 CRUD 用不到,但遇到特定场景(层级递归、排名分析、全文搜索、JSON 处理、冲突更新)能大幅简化代码。版本要求不一,多数需 SQLite 3.9+(2015)以上,JSON/窗口函数/UPSERT 需 3.25+/3.24+,基本现在的环境都满足。掌握这些能让 SQLite 远不止"轻量数据库",而是接近中型数据库的表达力。

#### CTE(with,含递归)

CTE(Common Table Expression)是用 `WITH` 定义的"临时命名结果集",本质是给子查询起个名字,让 SQL 更清晰可读。基本形态:`WITH 名 AS (SELECT ...) SELECT * FROM 名`,把复杂的子查询提到前面命名,主查询引用它,比嵌套子查询易读得多,还能在同一条 SQL 里定义多个 CTE 串联(用逗号分隔),层层加工数据。

真正强大的是**递归 CTE**:能遍历树形/层级数据,这是普通 SQL 做不到的。典型场景:组织架构(查某员工的所有下属)、物料 BOM(展开零件清单)、图的最短路径。写法是 `WITH RECURSIVE 名(列) AS (初始查询 UNION ALL 递归查询) SELECT ...`,初始查询给起点,递归查询引用自己逐步扩展,直到没有新数据产生。比如查层级:`WITH RECURSIVE 下属(id) AS (SELECT id FROM 员工 WHERE id=1 UNION ALL SELECT e.id FROM 员工 e JOIN 下属 d ON e.上级=d.id) SELECT * FROM 下属`--从 id=1 出发,一层层找出所有子孙。递归 CTE 是 SQL 处理层级数据的标准答案,以前得靠应用层递归或存储过程,现在一条 SQL 搞定。

#### 窗口函数(window functions)

窗口函数是数据分析的神器,能在不塌缩行的情况下做"跨行计算"--和聚合函数的区别是:聚合把多行合成一行(`GROUP BY` 后行数变少),窗口函数保留每行、只是给每行加一个"基于窗口的计算值"。语法是函数后跟 `OVER (PARTITION BY 列 ORDER BY 列)`,`PARTITION BY` 分窗(类似分组但行不塌)、`ORDER BY` 定义窗内顺序。

典型用途全靠几个核心函数:**`ROW_NUMBER()`** 给每行编号(分页、去重取最新);**`RANK()`/`DENSE_RANK()`** 排名(并列处理不同);**`LAG(列)`/`LEAD(列)`** 取前一行/后一行的值(算环比、算与上条订单差);**`SUM/AVG OVER`** 累计求和/移动平均(`SUM(金额) OVER (ORDER BY 日期)` 算累计销售额)。经典场景:每个客户最近一单(`ROW_NUMBER() OVER(PARTITION BY 客户id ORDER BY 下单时间 DESC)` 取 rn=1)、按赔付率排名、累计赔付率趋势。窗口函数在 SQLite 3.25+ 支持,做报表和精算统计时是降维打击,很多原本要导出 Excel 或写 Python pandas 的逻辑,一条 SQL 就能算。

#### FTS 全文检索

FTS(Full-Text Search)是 SQLite 的全文检索扩展,用来高效搜索文本内容--`LIKE '%关键词%'` 在大文本上慢且不能分词,FTS 建倒排索引支持中文分词(需配置)、模糊匹配、相关度排序。建 FTS 虚拟表:`CREATE VIRTUAL TABLE 文档 USING fts5(标题, 内容)`,插入数据正常插,查询用 `MATCH`:`SELECT * FROM 文档 WHERE 文档 MATCH '关键词'`,还能 `ORDER BY rank` 按相关度排序、用 `BM25()` 函数拿分数。

FTS 有 FTS3/FTS4/FTS5 三代,现在用 FTS5。价值在于:对文档、文章、聊天记录、商品描述这类大文本字段,做"包含某词"的搜索比 LIKE 快几个数量级,且支持分词、短语、前缀、布尔运算(`AND OR NOT`)。中文场景要额外配 tokenizer(如 simple、或第三方的 jieba tokenizer),否则按字切分效果一般。车险场景里,对通话录音转写文本、投诉记录做关键词检索,FTS 比 LIKE 实用得多。FTS 表是虚拟表,不存原数据(存倒排索引),常和主表配合用(主表存结构化数据,FTS 表存文本副本供检索)。

#### JSON 支持(json1 扩展)

SQLite 从 3.9 起内置 JSON 支持(json1 扩展,3.38+ 默认编译进核心),能在 SQL 里直接存取和查询 JSON。因为 SQLite 是动态类型,JSON 天然存为 TEXT,配合 JSON 函数解析。核心函数:`JSON_EXTRACT(列, '$.路径')` 取 JSON 字段值、`JSON_EACH` 把 JSON 数组展开成多行、`JSON_ARRAY_LENGTH` 数组长度、`JSON_INSERT/REPLACE/SET` 修改 JSON、`JSON_VALID` 判断是否合法 JSON。查询时还能用 `列 -> '$.字段'` 和 `->>` 运算符(3.38+)简写取值。

价值在于**半结构化数据**:有些字段结构不固定(如不同险种的附加属性、接口返回的多层嵌套),硬拆成列太僵化,存 JSON 灵活。配合 JSON 函数能在 SQL 层直接按 JSON 内字段过滤聚合(`WHERE JSON_EXTRACT(扩展, '$.渠道')='线上'`),不必把数据拽到应用层解析。生成列(见下)和 JSON 配合能"把 JSON 里常查的字段抽成实列加索引",兼顾灵活和性能。

#### UPSERT / 生成列(generated columns)

**UPSERT** 是 INSERT 的扩展,处理"主键/唯一键冲突时怎么办":`INSERT INTO t VALUES(...) ON CONFLICT(列) DO UPDATE SET ...`(冲突则更新)或 `DO NOTHING`(冲突则跳过)。等价于前面提过的 `INSERT OR REPLACE`/`INSERT OR IGNORE`,但 UPSERT(3.24+)语法更标准、更灵活(能精确指定冲突列和更新逻辑),是处理"存在则更新、不存在则插入"的规范写法,做数据同步、去重入库时极有用。

**生成列(generated columns)** 是"由其他列计算而来的列",定义时给出表达式,SQLite 自动维护,不能手动写。两种:`VIRTUAL`(用时现算,不存)和 `STORED`(存进库,占空间但读取快)。如 `总价 INTEGER GENERATED ALWAYS AS (数量*单价) VIRTUAL`。价值是把派生数据固化成列,可加索引、可查询,不用每次 SELECT 都重算。和 JSON 配合的经典用法:`数据 JSON, 渠道 TEXT GENERATED ALWAYS AS (JSON_EXTRACT(数据,'$.渠道')) STORED` 然后给"渠道"加索引--把 JSON 里常查的字段物化成可索引的列,兼顾 JSON 的灵活和索引的快速。

#### 串起来

CTE 让 SQL 清晰且能递归遍历层级;窗口函数在不塌行下做跨行分析,是报表统计神器;FTS 让大文本搜索告别 LIKE 的低效;JSON 支持让半结构化数据在 SQL 层可查可算;UPSERT 规范化"存在则更新";生成列把派生数据物化成可索引列。这些特性让 SQLite 远超"轻量数据库"的定位,很多原本要导出到 Python/pandas 或上 PostgreSQL 才能做的事,SQLite 原生就能搞定。版本要求基本现代环境都满足,知道有这些能力,遇到对应场景就不会 reinvent the wheel。
