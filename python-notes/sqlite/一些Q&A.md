# Q&A

**left join中，过滤条件写在 ON 和 WHERE 的区别是什么?**
“ON 是门，只拦右边人；WHERE 是筛，左右一起扔”
- 在 ON 里过滤：先过滤右表数据，再左连接。左表数据全保留，右表不匹配则显示 null。
- 在 WHERE 里过滤：先左连接生成大表，再过滤整行。如果右表字段为 null，会被 WHERE 条件剔除，导致 left join 退化成 inner join。


**覆盖索引是什么？**
查询所需所有列都在索引中，无需回表。极大提升性能。


**倒排索引是什么？**
- 正排索引:文档 → 词(给定一个文档,找出它包含哪些词)。这就是普通数据库存文本的
  方式--你有一条记录,要查它含不含某词,得把整段文本扫一遍。
- 倒排索引:词 →
  文档(给定一个词,直接找出包含它的所有文档)。方向反过来了,所以叫"倒排"。

倒排索引怎么建出来
  1. 分词:把每段文本切成词(中文要分词器,如"汽车保险便宜" ->
  ["汽车","保险","便宜"])
  2. 归一化:大小写统一、去停用词("的""了"等无意义词)、词干还原
  3. 建映射:统计每个词出现在哪些文档里,存成 词 -> 文档列表(这个列表叫 posting
  list)

一个例子秒懂

  三段文本:
  - Doc1:汽车保险便宜
  - Doc2:保险理赔快
  - Doc3:汽车便宜
  
  正排索引(每个文档列出它的词):
  Doc1 -> [汽车, 保险, 便宜]
  Doc2 -> [保险, 理赔, 快]
  Doc3 -> [汽车, 便宜] 
  
  倒排索引(每个词列出包含它的文档):
  汽车 -> [Doc1, Doc3]
  保险 -> [Doc1, Doc2]
  便宜 -> [Doc1, Doc3]
  理赔 -> [Doc2] 
  快   -> [Doc2]


**BM25算法是什么？**
相关性排序算法。TF-IDF的改进版，解决了TF-IDF的两个致命缺陷：
  1. 词频饱和问题。一个词出现100次和出现10次，在TF-IDF里线性增长，但实际相关性不会无限增加（比如”数据库“出现100次不代表比出现10次重要10倍）
  2. 文档长度偏差。长文档天然包含更多关键词，容易误判为更相关。
BM25通过非线性缩放和长度归一化，让评分更符合人类直觉。它是ES等搜索引擎等默认评分公式。
BM25的两个关键超参数：
  - k1（词频饱和度）
    - 调大：词频影响更大，适合强调关键词重复出现的场景
    - 调小：词频影响减弱，更看重“有没有”而不是“有多少”
  - b（长度惩罚系数）
    - 默认值：0.75
    - b=1：完全按长度归一化，长文档得分大幅降低
    - b=0：忽略长度，只关注词频和IDF（退化为类似TF-IDF）
    -  如果文档长度差异巨大，适当调大b；如果文档长度均匀，可保持 0.75
